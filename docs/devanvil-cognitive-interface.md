# DevAnvil Cognitive Interface

**Version:** 1.0  
**Status:** Canonical product philosophy (Hewn-wide)  
**Scope:** Signal, DevAnvil, and future Hewn products

---

## Purpose

Architect v3 revealed something deeper than a better architecture workflow.

Signal and DevAnvil are not two unrelated products. They are two instances of the **same cognitive process**.

- **Signal** externalizes creative thought.
- **DevAnvil** externalizes architectural thought.

This document names that pattern, makes it canonical, and defines the design constraints that follow.

The [Constitution](./devanvil-constitution.md) encodes the philosophy in Article IX.  
The [Workflow Model](./devanvil-workflow-model.md) implements it operationally.  
The Mental Model in Architect is its primary visualization today.

---

## The Cognitive Loop

Every complex domain follows the same progression:

```
Capture
  ↓
Reveal structure
  ↓
Reduce uncertainty
  ↓
Commit
  ↓
Execute
  ↓
Reflect
  ↓
Remember
```

The implementation changes. The cognition does not.

---

## Signal

```
Capture → Ground → Enrich → Bloom → Workbench → Runway
```

The Bloom is not the product.

**The Bloom is the visualization of understanding.**

---

## DevAnvil

```
Capture → Architect → Mental Model → Initiative → Forge → Evidence → Architectural Memory
```

The Mental Model is not documentation.

**It is the visualization of understanding.**

---

## Parallel Artifacts

| Stage | Signal | DevAnvil |
|-------|--------|----------|
| Capture | Frictionless idea | Inbox — preserve reality |
| Reveal structure | Ground, Enrich | Architect |
| Visualization | **Bloom** | **Mental Model** |
| Commit | Song / release decision | Initiative (model snapshot) |
| Execute | Workbench, production | Forge |
| Reflect | Creative reflection | Evidence |
| Remember | Creative Memory | Architectural Memory |

```
Signal:     Bloom → Song → Release → Creative Memory
DevAnvil:   Mental Model → Initiative → Implementation → Architectural Memory
```

The conversation is temporary. The visualization is the artifact.

---

## Shared Philosophy

Neither product should optimize for producing output.

They should optimize for **producing clarity**.

| Wrong optimization | Right optimization |
|--------------------|-------------------|
| More chat | Clearer structure |
| More documents | Living models |
| More tickets | Visible understanding |
| "AI built my feature" | "I understand my product now" |

**The artifact is never the chat.**

**The artifact is the user's understanding.**

## Bloom Was Never About Stars

The Bloom was never a constellation UI.

The Bloom was a way to **externalize uncertainty until it became understanding**.

DevAnvil's Mental Model is the same invention in a different language.

- One visualizes **creativity**.
- One visualizes **systems**.

---

## Architectural Thesis

DevAnvil should not become another AI coding assistant.

Its purpose is to become the place where **software architecture becomes visible**.

Users should stop thinking:

> "I'm asking AI questions."

and instead feel:

> "I'm watching my understanding evolve."

---

## From Documents to Living Models

Current software tools produce documents:

- Requirements
- Tickets
- Design docs
- Architecture diagrams

DevAnvil should instead produce **living models**.

Every conversation changes the model.  
Every decision strengthens relationships.  
Every uncertainty becomes explicit.  
Every assumption either becomes a decision or is rejected.

The architecture is alive until implementation begins.

---

## Mental Models Are the Primary Artifact

| Layer | Role |
|-------|------|
| **Mental Model** | Primary artifact — permanent understanding |
| **Conversation** | Temporary — exists only to improve the model |
| **Initiative** | Snapshot of the model at commit time |
| **Forge** | Executes what the model decided |
| **Evidence** | Validates the model against reality |
| **Memory** | Preserves the evolution of the model |

Initiatives must emerge from the model — never from echoed conversation.

---

## Model Semantics (v3)

A Mental Model makes architecture visible through:

- **Nodes** — domains, concepts, subsystems, capabilities
- **Relationships** — with per-link confidence
- **Assumptions** — locked (✓) vs open (?)
- **Competing options** — architectures compared, not argued in prose
- **Changes** — visible evolution each turn (new node, boundary moved, split)
- **Open questions** — only unresolved architectural unknowns

Confidence attaches to **beliefs** (nodes and relationships), not to the session.

---

## The Next Evolution

Today's Mental Model is still textual.

Eventually it should become **spatial**:

- Nodes and edges
- Boundaries that move
- Confidence that strengthens visually
- Assumptions that collapse into decisions
- Competing models side by side

As discussion progresses:

- relationships strengthen
- nodes move
- boundaries emerge
- confidence grows
- assumptions collapse into decisions

**The architecture literally settles.**

Exactly as a Bloom settles.

---

## Design Constraints

Do not rush toward a visual canvas because Bloom has one.

Instead:

1. **Make the Mental Model indispensable.**
2. **Make every conversation improve it.**
3. **Make initiatives emerge from it naturally.**
4. **Let the correct visualization reveal itself over time.**

Bloom taught us this lesson already.

The right visualization emerged because the **underlying thinking model was correct**.

Architect should evolve the same way.

---

## Success Criteria

DevAnvil succeeds when founders open it **before they write code** — not because they need tickets, but because they need clarity.

The highest compliment is not:

> "AI built my feature."

It is:

> "I understand my product now."

That is the same feeling artists have when a Bloom finally settles.

That is the feeling every Hewn product should strive to create.
